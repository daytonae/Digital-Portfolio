Database Design & Modeling

ER Diagrams & Relational Schema

Throughout a good chunk of semester my team and I focused on ER diagrams and relational schemas. For our project "FLOW", we designed diagrams and schemas that follow a restaurant management system with entities such as person, customer, employee, orders, menu_item, and inventory. These represented real word restaurant operations such as customer activity, employee roles, and order processing.

As a team, we agreed on this scenario early on, which actually made the entire design process a lot more engaging. SQL Implementation

DDL, Queries & Index Analysis

This section highlights my use of SQL for defining, querying, and analyzing relational data. Through Lab 1, generating and practicing SQL problems, and the development of the FLOW system, I created tables using DDL while incorporating primary keys, foreign keys, and constraints to enforce data integrity within the database.

Additionally, I developed SQL queries to retrieve and manipulate data using commands such as SELECT, WHERE, and JOIN. These queries allowed for efficient data retrieval and made it possible to connect related tables.

Advanced Database Concepts

Transactions, Concurrency & Optimization

I learned about transactions and concurrency and how they affect database operations in real systems. Transactions ensure that a group of operations is either fully completed or not executed at all, helping maintain consistency and prevent partial updates.

For instance, within FLOW, placing an order may involve inserting an order record and updating related data (such as ingredient counts). If one part fails, the entire transaction should be rolled back to prevent incomplete or incorrect data.

I also explored concurrency, which allows multiple users or processes to interact with the database at the same time. For example, multiple customers placing orders across different branch locations requires the database to handle a high volume of activity without conflicts or errors. These concepts helped me understand how databases remain reliable and stable under real-world conditions.

NoSQL Concepts

MongoDB Document Modeling

Regarding NoSQL concepts such as MongoDB document modeling, I explored it as an alternative to relational database design. Unlike relational databases, MongoDB stores data in flexible, JSON-like documents rather than structured tables. this allows related data to be grouped together within a single document instead of being spread across multiple tables.

For instance, within FLOW, customer reviews could be stored as documents that include user info, ratings, and comments all in one place instead of multiple tables.

This helped me understand the trade-off between structure and flexibility, and deciphering when a NoSQL approach may be more efficient depending on the type of data being stored.

Labs & Exercises

Redis, In-Memory & Hands-On Labs

I explored Redis as part of our NoSQL database presentation, where we examined how in-memory databases differ from traditional relational systems and when they should be used. Unlike SQL databases, Redis stores data in memory, which allows for much faster data access and retrieval.

As a team, we chose to incorporate Redis concepts into the FLOW system to better understand how NoSQL can complement relational databases. Redis is commonly used for caching and real-time data access, where speed is critical. For example, Redis can be used within the system to cache frequently accessed data such as menu items, which can then populate features like a "popular" category without repeatedly querying the database.

Overall, this helped me understand the benefits of in-memory databases, as well as the trade-offs, such as limited persistence and higher memory usage.

Project Evidence — FLOW

Soul by the Sea Restaurant Database System
System Overview

What FLOW Is

FLOW is an enterprise-level restaurant management system designed to support operations across multiple branch locations. Its purpose is to centralize key business functions such as POS transactions, reservations, inventory tracking, supplier management, workforce scheduling, and customer feedback within a single platform. The name "FLOW" reflects the systems design goal of allowing data to move seamlessly across different components, supporting efficient and integrated operations.

The system is built around core entities including person, customer, employee, orders, menu_item, inventory_item, supplier, and reservation, which all model real-world restaurant operations. These entities support workflows such as order processing, staff management, inventory update, and customer interactions across different branches.

Scope: FLOW is designed to handle both operational tasks and analytics. managers can compare branch performance, monitor sales trends, and optimize staffing and menu decisions, while staff and customers interact with the system through dedicated user interfaces.

Architecture: FLOW uses a relational database (MySQL) to manage structures and transactional data, ensuring consistency. NoSQL concepts are included for the handling of more flexible data such as customer reviews and analytics-related information. This hybrid approach allows the system to balance structure, scalability, and performance in a realistic environment.

Database Design Decisions

Schema, Keys & Normalization

Database Platform & Configuration

FLOW uses MySQL as the primary database that manages structured and transactional data. Functions such as orders, reservations, inventory management, and employee records are stored within relational tables. These require reliable data handling, especially given the multi-branch environment where multiple processes may occur simultaneously.

The relational schema is designed using primary keys, foreign keys, and constraints to enforce data integrity and maintain the correct relationships between entities. For instance:

  • Each order must reference a customer, employee, and branch, ensuring that all transactions are tied to valid system records.
  • Every reservation is linked to a specific customer and branch location, preventing bookings from existing without correct context
  • Inventory records are connected to both menu items and suppliers, ensuring that stock levels stay consistent
  • Payments are associated with a specific order, therefore every financial transaction is linked to a valid purchase.
  • Employees are linked to a branch location, allowing for correct management of scheduling, staffing, and operational responsibilities
  • Order items are tied to both an order and a menu items

In addition, the system supports multiple input types such as customer names, menu items, reviews, and more. This is important because system inputs ma vary, such as special characters in names or custom menu descriptions.

Overall, the database platform and config choices prioritize reliability, consistency, and structured data management, forming the foundation for the system.

Normalization to 3NF

The normalization process for the FLOW system was completed in stages, beginning with identifying redundancies in the initial design process. At first, there were separate entities created for different types of people such as customer, employee, supplier, manager. However, we noticed repeated attributes such as names, phone numbers, and email addresses across multiple tables. To address this, the schema was reconstructed by introducing a person super type, which just stores all shared attributes. Subtype tables such as customer, employee, and manager were then create to store role-specific information.

Further normalization steps focused on separating data into the correct tables based on functional dependencies. For example, order details were split between orders and order_items to ensure each table stores data directly related to its primary key. The same can be said with with menu and menu_items, inventory, and suppliers.

Throughout this entire process, the schema was continuously analyzed to ensure that all non key attributes depended only on their primary keys without partial dependencies. This allowed the database to meet the requirements of 3NF, improving data consistency and reducing any risk of anomalies during updates, insertions, or deletions.

The final table structure reflects these normalization decision by organizing data into tables connected through primary and foreign key relationships. This structure makes the data easier to manage, maintain, and scale while keeping it accurate and consistent.

Key Constraints & ENUMs

The FLOW system uses primary and foreign keys to keep data connected and consistent. Foreign keys link related tables, such as orders to customers, employees, and branches, ensuring that records cannot exist without valid references. This prevents issues like invalid or incomplete data.

For instance, as previously stated, each order must reference a valid person and branch, while payments are tied to a specific order. Reservations are also linked to both a customer and a branch to maintain the correct relationships across the system.

Additionally, certain fields are restricted to specific values consistency. For instance, fields such as order status and reservation only allow predefined values such as "IN_PROGRESS", "SERVED", or "CANCELLED". Reservation times are also limited to branch hours only.

Additional constraints such as NOT NULL, UNIQUE, and CHECK conditions are used to ensure required data is present and values stay valid.

Data Flow & System Integration

Hybrid Architecture

FLOW uses a hybrid architecture that combines MySQL, Redis, and MongoDB to handle different types of data. MySQl us used fr structured, transactional data such as orders, reservations, inventory, employee records where consistency is a key factor. Redis is used as a caching layer to improve performance for frequently accessed data such as populating the "popular" category on the website based off the most ordered items. MongoDB is used for flexible data such as feedback and reviews/analytics.

Data in the system follows a flow from user interface to the database layer. When a user performs an action, such as placing an order, the request is first handled by the Python application layer. The application validates the input and determines how the data should be processed. Before querying, Redis may be checked for cached data to reduce unnecessary database calls and improve response times.

If data is not cached, the system interacts with MySQL to handle it. For example, placing an order involves inserting records into the orders table and updating related inventory data. his ensures accuracy and consistency.

After the transaction is processed, relevant event or tracking data can be sent to MongoDB where it is stored in a more flexible format for use in analytics, logging, or real-time updates. This allows the system to handle both structured and unstructured data.

Overall, this hybrid approach allows FLOW to maintain strong data integrity for core operations while also supporting flexibility and performance, making it suitable for a real-world, multi-branch system.

Scalability & Performance Considerations

Redis Caching & Optimization

Redis is used in the FLOW system as a caching layer to improve performance and reduce unnecessary database queries. Before sending requests to the main database, the system first checks Redis for frequently accessed data such as menu items or dashboard information. If the data is found in cache, it is returned immediately, which speeds up response time. If not, the system queries MySQL and stores the result in Redis for future use. This reduces repeated database calls and helps the system handle higher traffic more efficiently.

SQL vs. NoSQL vs. NewSQL/Graph

In FLOW, MySQL stores core transactional data such as orders, inventory, reservations, and employee records. MongoDB stores flexible data like customer feedback and Redis is used to cache frequently accessed data such as menu items.

These choices reflect trade-offs in system design. MySQL is used where consistency matters the most, MongoDB where flexibility is needed, and Redis where speed is a priority. Based on this, I would choose between these systems in a real project by matching the database to the type of data.

Design Trade-Offs

One major architecture decision was switching the Customer UI from Tkinter to a website. Since customers would realistically expect a web-based interface, using Flask for the customer size made the system easier to access, easier to demonstrate, and more engaging for the class. However, this also created a trade-off because the project now has both a web interface and python desktop interface to manage.

Another trade-off was balancing normalization with usability. Normalizing the schema helped reduce duplicate data, especially with the person supertype and customer/employee subtypes, but it also made some queries more complex because related data and to be joined across multiple tables. For MongoDB, flexible data such as reviews and even logs worked better as documents, while core relationships stayed in MySQL. In hindsight, i would probably spend more time planning indexes and data access patterns earlier, especially for branch based reports, menu searches, and frequently accessed data that could benefit from Redis caching.
